# Руководство пользователя

## IMAP Mail Migration Tool (imap_migrate)

Версия 1.0.0

---

## 1 Введение

### 1.1 Назначение документа

Настоящее руководство описывает порядок установки, настройки и использования инструмента командной строки IMAP Mail Migration Tool (далее --- imap_migrate). Документ предназначен для пользователей, выполняющих миграцию электронной почты между IMAP-серверами.

### 1.2 Назначение системы

imap_migrate --- инструмент командной строки для переноса электронной почты между произвольными IMAP-серверами. Поддерживает возобновление прерванной миграции, дедупликацию сообщений по Message-ID и кэшу UID, управление паузой через сигналы и файл-флаг, автоматическое переподключение при сбоях сети.

Типичные сценарии использования:

- перенос почты при смене почтового провайдера (Gmail, Yandex, Mail.ru, корпоративный сервер);
- миграция на self-hosted сервер;
- резервное копирование почты на другой сервер;
- параллельный перенос нескольких почтовых ящиков.

### 1.3 Условия применения

Для работы инструмента требуется:

- Python 3.10 или выше;
- доступ к исходному и целевому IMAP-серверам по сети;
- учётные данные (логин и пароль/пароль приложения) для обоих серверов;
- операционная система: macOS, Linux или Windows.

### 1.4 Технологический стек

| Компонент | Технология |
|-----------|-----------|
| Язык | Python 3.10+ |
| Протокол | IMAP4rev1 (imaplib) |
| Конфигурация | YAML (PyYAML) или JSON |
| Индикатор прогресса | tqdm |
| Запуск | Bash (run.sh) |

## 2 Установка и подготовка к работе

### 2.1 Получение дистрибутива

Клонируйте репозиторий или скачайте архив с исходным кодом. Перейдите в каталог проекта:

```bash
cd imap_migrate
```

### 2.2 Создание конфигурации

Скопируйте шаблон конфигурации и откройте его в текстовом редакторе:

```bash
cp config.yaml.example config.yaml
nano config.yaml
```

Подробное описание параметров приведено в разделе 3.

### 2.3 Запуск через run.sh (рекомендуется)

Скрипт `run.sh` автоматически создаёт виртуальное окружение Python и устанавливает зависимости при первом запуске:

```bash
chmod +x run.sh
./run.sh
```

При последующих запусках виртуальное окружение используется повторно.

### 2.4 Ручная установка

При необходимости можно создать виртуальное окружение вручную:

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

После активации окружения инструмент запускается напрямую:

```bash
python imap_migrate.py --config config.yaml --list-folders
```

### 2.5 Удаление виртуального окружения

Для очистки виртуального окружения используйте команду:

```bash
./run.sh clean
```

## 3 Конфигурация

Конфигурация хранится в файле формата YAML (по умолчанию `config.yaml`). Путь к файлу задаётся переменной окружения `IMAP_CONFIG` или параметром `--config`.

### 3.1 Параметры исходного сервера (source)

| Параметр | Тип | Обязательный | По умолчанию | Описание |
|----------|-----|:------------:|:------------:|----------|
| `host` | строка | да | --- | Имя хоста IMAP-сервера |
| `port` | целое | нет | 993 | Порт подключения |
| `user` | строка | да | --- | Имя пользователя (email) |
| `password` | строка | да | --- | Пароль или пароль приложения |
| `ssl` | логический | нет | true | Использовать SSL/TLS |
| `starttls` | логический | нет | false | Использовать STARTTLS вместо SSL |

### 3.2 Параметры целевого сервера (destination)

Аналогичны параметрам исходного сервера (раздел 3.1).

### 3.3 Параметры миграции (options)

| Параметр | Тип | По умолчанию | Описание |
|----------|-----|:------------:|----------|
| `timeout` | целое | 120 | Тайм-аут подключения (секунды). Увеличьте для медленных серверов |
| `throttle` | дробное | 0.05 | Задержка между сообщениями (секунды). Предотвращает перегрузку сервера |
| `max_retries` | целое | 3 | Максимум попыток отправки одного сообщения |
| `batch_limit` | целое | 0 | Лимит сообщений за один запуск (0 --- без лимита) |
| `scan_batch_size` | целое | 100 | Размер пакета при сканировании Message-ID |
| `folder_retries` | целое | 3 | Повторные попытки обработки папки при критической ошибке |
| `noop_interval` | целое | 30 | Интервал отправки NOOP для поддержания соединения (секунды) |
| `reconnect_max_wait` | целое | 300 | Максимальное время ожидания при переподключении (секунды) |
| `status_interval` | целое | 600 | Интервал вывода прогресса в лог (секунды, 0 --- отключить) |
| `use_builtin_map` | логический | true | Использовать встроенную карту имён папок |
| `verbose` | логический | false | Включить отладочное логирование (уровень DEBUG) |
| `state_file` | строка | авто | Имя файла состояния (по умолчанию генерируется из имени конфига) |
| `log_file` | строка | авто | Имя файла логов |
| `exclude_flags` | список | [] | Флаги сообщений, исключаемые при переносе |

### 3.4 Управление папками

#### Встроенная карта папок

При `use_builtin_map: true` (по умолчанию) инструмент автоматически преобразует локализованные имена папок в стандартные IMAP-имена:

| Исходное имя | Целевое имя |
|-------------|-------------|
| Входящие | INBOX |
| Отправленные, [Gmail]/Sent Mail | Sent |
| Черновики, [Gmail]/Drafts | Drafts |
| Спам, [Gmail]/Spam | Junk |
| Удалённые, Корзина, [Gmail]/Trash | Trash |
| Outbox | Outbox |

#### Пользовательская карта папок

Для переопределения маппинга укажите секцию `folder_map`:

```yaml
folder_map:
  "Входящие": "INBOX"
  "My Custom Folder": "CustomDest"
```

#### Фильтрация папок

Перенос только указанных папок:

```yaml
only_folders:
  - INBOX
  - Sent
  - Drafts
```

Исключение папок из переноса:

```yaml
exclude_folders:
  - Spam
  - Trash
  - "[Gmail]/Important"
```

### 3.5 Примеры конфигурации для провайдеров

#### Yandex Mail

```yaml
source:
  host: imap.yandex.ru
  port: 993
  user: username@yandex.ru
  password: <пароль-приложения>
  ssl: true
```

Для генерации пароля приложения перейдите по адресу: <https://id.yandex.ru/security/app-passwords>, выберите тип «Почта».

#### Gmail

```yaml
source:
  host: imap.gmail.com
  port: 993
  user: user@gmail.com
  password: <пароль-приложения>
  ssl: true
```

Пароль приложения создаётся в настройках Google: Аккаунт Google > Безопасность > Пароли приложений.

#### Mail.ru

```yaml
source:
  host: imap.mail.ru
  port: 993
  user: user@mail.ru
  password: пароль
  ssl: true
```

#### Self-hosted (STARTTLS)

```yaml
source:
  host: mail.yourdomain.com
  port: 143
  user: user@yourdomain.com
  password: пароль
  ssl: false
  starttls: true
```

## 4 Команды и параметры запуска

### 4.1 Команды через run.sh

| Команда | Описание |
|---------|----------|
| `./run.sh list-folders` | Показать папки на исходном сервере с количеством сообщений |
| `./run.sh dry-run` | Имитация миграции без записи на целевой сервер |
| `./run.sh run` | Выполнить миграцию |
| `./run.sh verify` | Миграция с последующей проверкой количества сообщений |
| `./run.sh clean` | Удалить виртуальное окружение .venv |

### 4.2 Дополнительные параметры

| Параметр | Краткая форма | Описание |
|----------|:-------------:|----------|
| `--config` | `-c` | Путь к файлу конфигурации (обязателен при прямом запуске) |
| `--list-folders` | --- | Вывести список папок и выйти |
| `--dry-run` | `-n` | Тестовый прогон без записи |
| `--folders` | `-f` | Перенести только указанные папки (через запятую) |
| `--verify` | --- | Проверить количество сообщений после миграции |
| `--verbose` | `-v` | Включить отладочный уровень логирования |

### 4.3 Переменная окружения IMAP_CONFIG

Для указания пути к конфигурации без параметра `--config`:

```bash
IMAP_CONFIG=/path/to/config.yaml ./run.sh run
```

### 4.4 Примеры запуска

Просмотр папок на исходном сервере:

```bash
./run.sh list-folders
```

Тестовый прогон (без записи):

```bash
./run.sh dry-run
```

Миграция всех папок:

```bash
./run.sh run
```

Миграция только INBOX и Sent:

```bash
./run.sh run --folders INBOX,Sent
```

Миграция с проверкой:

```bash
./run.sh verify
```

Отладочный режим:

```bash
./run.sh run --verbose
```

## 5 Порядок работы

### 5.1 Первый запуск

1. Создайте конфигурацию из шаблона (раздел 2.2).
2. Заполните учётные данные исходного и целевого серверов.
3. Проверьте доступность серверов и корректность учётных данных:
   ```bash
   ./run.sh list-folders
   ```
   Команда выведет список папок с количеством сообщений и маппингом имён.
4. Выполните пробный прогон:
   ```bash
   ./run.sh dry-run
   ```
   Убедитесь, что ошибок нет, маппинг папок корректен.
5. Запустите миграцию:
   ```bash
   ./run.sh run
   ```

При запуске инструмент выводит таблицу обзора папок с информацией о количестве сообщений на источнике и приёмнике, объёме данных и прогрессе:

```
Обзор папок:
Папка                          Src     Src MB    Dst   Сост  Передано  Осталось  Данных
────────────────────────────────────────────────────────────────────────────────────────
INBOX → INBOX              180 256   82.3 GB  79 420  124 407  66.4 GB   55 849  15.9 GB
Sent → Sent                  2 341    1.2 GB   2 341    2 341   1.2 GB        0     0 B
────────────────────────────────────────────────────────────────────────────────────────
Итого                      182 597   83.5 GB  81 761  126 748  67.6 GB   55 849  15.9 GB
```

### 5.2 Возобновление прерванной миграции

При повторном запуске инструмент автоматически загружает файл состояния и продолжает миграцию с места остановки:

```bash
./run.sh run
```

Механизм возобновления:

- загружается файл `migration_state_<config>.json`;
- ранее перенесённые сообщения пропускаются по Message-ID;
- если UIDVALIDITY папки не изменился, используется кэш UID для быстрого пропуска без повторного FETCH;
- индикатор прогресса показывает оставшиеся сообщения.

### 5.3 Перенос отдельных папок

Для выборочного переноса укажите папки через параметр `--folders`:

```bash
./run.sh run --folders INBOX,Sent,Drafts
```

Альтернативно задайте фильтрацию в конфигурации (раздел 3.4).

### 5.4 Миграция нескольких почтовых ящиков

Создайте отдельный конфигурационный файл для каждого ящика:

```bash
cp config.yaml.example user1.yaml
cp config.yaml.example user2.yaml
```

Запустите миграцию для каждого ящика:

```bash
IMAP_CONFIG=user1.yaml ./run.sh run
IMAP_CONFIG=user2.yaml ./run.sh run
```

Каждый конфигурационный файл получает изолированные файлы состояния и логов. Допускается параллельный запуск миграций разных ящиков.

### 5.5 Проверка результатов

После миграции выполните проверку:

```bash
./run.sh verify
```

Инструмент подключится к обоим серверам, сравнит количество сообщений по папкам и выведет отчёт о совпадениях и расхождениях. При обнаружении расхождений рекомендуется повторно запустить миграцию.

## 6 Управление процессом миграции

### 6.1 Пауза и возобновление

#### Способ 1: Ctrl+C (все платформы)

Однократное нажатие Ctrl+C:

- завершает перенос текущего сообщения;
- сохраняет состояние на диск;
- корректно завершает процесс.

Повторное нажатие Ctrl+C:

- немедленное принудительное завершение.

Для возобновления запустите ту же команду повторно.

#### Способ 2: Сигнал SIGUSR1 (macOS, Linux)

При запуске инструмент выводит PID процесса:

```
PID: 12345 (kill -USR1 12345 для паузы)
```

Для приостановки отправьте сигнал:

```bash
kill -USR1 12345
```

Процесс приостанавливается, состояние сохраняется. Соединения поддерживаются командами NOOP.

Для возобновления отправьте тот же сигнал повторно:

```bash
kill -USR1 12345
```

#### Способ 3: Файл-флаг (Windows и другие платформы)

Создайте пустой файл `.migration.pause` в рабочем каталоге:

```bash
touch .migration.pause
```

Процесс обнаружит файл и приостановится. Для возобновления удалите файл:

```bash
rm .migration.pause
```

### 6.2 Блокировка параллельного запуска

Инструмент создаёт файл блокировки `migration_state_<config>.lock` при запуске. Попытка запустить вторую копию с тем же конфигурационным файлом приведёт к ошибке. Файл блокировки автоматически удаляется при завершении процесса.

Если процесс завершился аварийно и файл блокировки не был удалён, убедитесь, что процесс не запущен, и удалите файл вручную:

```bash
rm migration_state_<config>.lock
```

## 7 Файлы, создаваемые при работе

### 7.1 Файл состояния

**Файл:** `migration_state_<config>.json`

Содержит:

- `migrated` --- множество Message-ID перенесённых сообщений по папкам;
- `uid_cache` --- кэш UID источника для быстрого пропуска;
- `uidvalidity` --- значения UIDVALIDITY для контроля актуальности кэша;
- `folder_stats` --- статистика по папкам (количество и объём).

Файл обновляется после каждого пакета сообщений и при остановке. Запись выполняется атомарно через временный файл `.tmp`.

### 7.2 Файл логов

**Файл:** `migration_<config>.log`

Содержит хронологическую запись всех действий с метками времени и уровнями:

- `INFO` --- штатные операции, прогресс;
- `WARNING` --- переподключения, нефатальные ошибки;
- `ERROR` --- ошибки при переносе отдельных сообщений;
- `DEBUG` --- детализированная трассировка (при `--verbose`).

### 7.3 Файл блокировки

**Файл:** `migration_state_<config>.lock`

Предотвращает параллельный запуск с одним конфигурационным файлом. Создаётся при старте, удаляется при завершении.

### 7.4 Файл паузы

**Файл:** `.migration.pause`

Создаётся пользователем вручную для приостановки миграции (раздел 6.1).

### 7.5 Конфигурация

**Файл:** `config.yaml`

Содержит учётные данные серверов. Не включается в систему контроля версий.

## 8 Устранение неполадок

### 8.1 Ошибки аутентификации

**Сообщение:** `Login failed` или `[AUTHENTICATIONFAILED]`

Возможные причины и решения:

1. **Используется обычный пароль вместо пароля приложения.** Gmail, Yandex и некоторые другие провайдеры требуют специальный пароль приложения для IMAP-доступа. Создайте пароль приложения в настройках безопасности аккаунта.
2. **IMAP-доступ отключён.** Проверьте настройки почтового ящика у провайдера и включите доступ по протоколу IMAP.
3. **Некорректные учётные данные.** Проверьте правильность логина и пароля в конфигурационном файле.

### 8.2 Ошибки подключения

**Сообщения:** `Broken pipe`, `EOF`, `Connection reset by peer`

Возможные причины и решения:

1. **Тайм-аут сети.** Увеличьте значение `timeout` в конфигурации (например, до 240--300 секунд).
2. **Перегрузка сервера.** Уменьшите `scan_batch_size`, увеличьте `throttle`.
3. **Высокая задержка сети.** Увеличьте `timeout` и `reconnect_max_wait`.

Инструмент автоматически переподключается с экспоненциальной задержкой (2, 4, 8 секунд и далее до `reconnect_max_wait`).

### 8.3 Ошибки папок

**Сообщение:** `Папка '...' недоступна`

Решения:

- проверьте права доступа к папке на целевом сервере;
- создайте папку вручную, если автоматическое создание не удалось;
- при проблемах с именами папок используйте `folder_map` для явного маппинга.

### 8.4 Повреждение файла состояния

**Сообщение:** `State-файл повреждён. Миграция начнётся заново.`

Инструмент пытается восстановить состояние из резервной копии `.tmp`. Если восстановление невозможно, миграция начинается заново. Ранее перенесённые сообщения будут пропущены по дедупликации Message-ID на стороне сервера, однако повторное сканирование займёт время.

### 8.5 Расхождения при проверке

Если после `./run.sh verify` обнаружены расхождения в количестве сообщений:

1. Проверьте лог-файл на наличие ошибок.
2. Повторно запустите миграцию --- инструмент перенесёт оставшиеся сообщения.
3. Повторите проверку.

Типичные причины расхождений: использование `batch_limit`, ошибки при переносе отдельных сообщений, сообщения без заголовка Message-ID.

### 8.6 Низкая производительность

Рекомендации по ускорению:

- увеличьте `scan_batch_size` (больше сообщений за одно обращение к серверу);
- уменьшите `throttle` (меньше задержка между сообщениями);
- при стабильном соединении уменьшите `timeout`;
- проверьте качество сети: `ping imap.source.com`.

## 9 Безопасность

### 9.1 Хранение учётных данных

Конфигурационный файл содержит пароли в открытом виде. Рекомендации:

- установите права доступа `chmod 600 config.yaml` (только владелец);
- не добавляйте файл в систему контроля версий (файл включён в `.gitignore`);
- после завершения миграции удалите или обезопасьте конфигурацию.

### 9.2 Маскировка паролей

В логах и трассировках стека пароли заменяются на `***`. Объект `ServerConfig.__repr__` маскирует поле `password`.

### 9.3 Сетевая безопасность

По умолчанию используется SSL/TLS-соединение (порт 993). Для серверов, не поддерживающих прямой SSL, доступен режим STARTTLS.

## 10 Ограничения

1. Инструмент переносит только стандартные IMAP-флаги (`\Seen`, `\Answered`, `\Flagged`, `\Deleted`, `\Draft`). Пользовательские флаги и метки (например, метки Gmail) не переносятся.
2. Флаг `\Recent` не переносится, так как он устанавливается сервером автоматически.
3. Структура вложенных папок сохраняется, но права доступа и подписки определяются целевым сервером.
4. Сообщения без заголовка Message-ID отслеживаются по UID и могут дублироваться при изменении UIDVALIDITY.
5. Максимальный размер сообщения ограничен настройками целевого IMAP-сервера.

## Приложение А. Структура каталога проекта

```
imap_migrate/
├── imap_migrate.py             # Основной инструмент CLI
├── run.sh                      # Скрипт запуска
├── requirements.txt            # Зависимости Python
├── config.yaml.example         # Шаблон конфигурации
├── config.yaml                 # Конфигурация (не в Git)
├── .venv/                      # Виртуальное окружение (создаётся run.sh)
├── migration_state_*.json      # Файл состояния
├── migration_state_*.lock      # Файл блокировки
├── migration_*.log             # Файл логов
└── .migration.pause            # Файл-флаг паузы (создаётся вручную)
```
