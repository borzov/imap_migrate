# IMAP Mail Migration

Скрипт переносит почту между **любыми IMAP-серверами**: Яндекс, Gmail, Mail.ru, корпоративная почта, свой сервер. Умеет докачивать с места остановки (в т.ч. по UID-кэшу — без повторного сканирования уже перенесённых писем), ставить на паузу и возобновлять, переподключаться при обрывах и при временной потере сети. Сохраняет стандартные флаги и даты писем. Подходит для длительных миграций с большими ящиками.

История изменений: [CHANGELOG.md](CHANGELOG.md).

## Что понадобится

- Python 3.10+
- Учётные данные источника и назначения (логин + пароль). Для Яндекса и Gmail обычно нужен **пароль приложения**, а не основной пароль.
- Включённый доступ по IMAP в настройках почты источника и приёмника.

## Настройка

Конфиг с паролями в репозиторий не коммитится. Создайте свой конфиг из примера:

```bash
cd imap_migrate
cp config.yaml.example config.yaml
nano config.yaml   # или любой редактор
```

В `config.yaml` обязательно заполните:

- **source:** `host`, `user`, `password` — IMAP-сервер, с которого переносим (например `imap.yandex.ru`, `imap.gmail.com`, `imap.mail.ru`).
- **destination:** `host`, `user`, `password` — целевой IMAP-сервер.

Остальные параметры в `options` можно оставить по умолчанию. Полезные опции: `timeout`, `throttle`, `status_interval` (интервал вывода прогресса в лог), `use_builtin_map` (автомаппинг имён папок).

### Провайдеры (источник или назначение)

- **Яндекс** — `host: imap.yandex.ru`, порт 993. Пароль приложения: [id.yandex.ru/security/app-passwords](https://id.yandex.ru/security/app-passwords) (раздел «Почта»).
- **Gmail** — `host: imap.gmail.com`, порт 993. Включите IMAP в настройках Gmail и создайте пароль приложения в аккаунте Google.
- **Mail.ru** — `host: imap.mail.ru`, порт 993. В настройках почты включите доступ по IMAP.
- **Свой сервер** — укажите `host` и порт (обычно 993 для SSL).

Встроенный маппинг папок (`use_builtin_map: true`) переводит русские и типичные имена (Входящие, Sent, [Gmail]/Trash и т.д.) в стандартные IMAP-имена (INBOX, Sent, Trash, Junk). При необходимости свой маппинг задаётся в `folder_map`.

## Перенос нескольких ящиков

Для переноса нескольких ящиков создайте отдельный конфиг для каждого:

```bash
cp config.yaml.example user1.yaml  # настройте user1
cp config.yaml.example user2.yaml  # настройте user2

./run.sh run -c user1.yaml   # state: migration_state_user1.json
./run.sh run -c user2.yaml   # state: migration_state_user2.json
```

Каждый конфиг автоматически получает собственный state-файл (`migration_state_<name>.json`) и лог (`migration_<name>.log`) — данные разных ящиков не смешиваются.

Запускать два процесса с одним конфигом одновременно нельзя — скрипт обнаружит это и завершится с ошибкой (защита от случайного дублирования).

**Если у вас уже есть `migration_state.json` от предыдущей версии:** переименуйте его под новое имя:

```bash
mv migration_state.json migration_state_config.json
```

## Запуск

Удобнее всего через `run.sh`: он поднимет venv и подставит нужный конфиг.

```bash
chmod +x run.sh
./run.sh
```

При первом запуске создаётся виртуальное окружение и ставятся зависимости. Дальше:

```bash
# Посмотреть папки на источнике
./run.sh list-folders

# Пробный прогон без записи
./run.sh dry-run

# Сам перенос
./run.sh run

# Перенос и потом сверка количества писем по папкам
./run.sh verify
```

По умолчанию используется `config.yaml` из текущей папки. Если конфиг в другом месте:

```bash
IMAP_CONFIG=/path/to/config.yaml ./run.sh run
```

Только отдельные папки:

```bash
./run.sh run --folders INBOX,Sent
```

## Обзор папок при запуске

При каждом запуске (в том числе `dry-run` и при возобновлении) скрипт выводит таблицу по всем обрабатываемым папкам:

```
  Обзор папок:
  Папка                                    Src     Src MB      Dst     Сост Передано  Осталось   Данных
  ──────────────────────────────────────────────────────────────────────────────────────────────────
  INBOX → INBOX                        180 256   82.3 GB   79 420  124 407  66.4 GB    55 849  15.9 GB
  Sent → Sent                            2 341    1.2 GB    2 341    2 341   1.2 GB         0     0 B
  ──────────────────────────────────────────────────────────────────────────────────────────────────
  Итого                                182 597   83.5 GB   81 761  126 748  67.6 GB    55 849  15.9 GB
```

Колонки:
- **Src** — число писем на источнике.
- **Src MB** — суммарный размер писем на источнике (только метаданные, тела не загружаются).
- **Dst** — число писем на приёмнике.
- **Сост** — уже перенесено согласно state-файлу (без обращения к серверу).
- **Передано** — объём данных из state-файла.
- **Осталось** — сколько писем / данных ещё не перенесено; подсвечивается зелёным при 0.

## Управление процессом

- **Остановка (Ctrl+C)** — корректное завершение: текущее письмо дописывается, состояние сохраняется в `migration_state_<config>.json`. Повторный запуск продолжит с того же места. Второй Ctrl+C — немедленный выход.
- **Пауза (macOS/Linux)** — при старте в логе выводится PID процесса. Команда `kill -USR1 <PID>` ставит миграцию на паузу; повторная отправка того же сигнала возобновляет работу. Состояние при паузе сохраняется.
- **Пауза (Windows)** — сигнал USR1 недоступен. Создайте в каталоге с `migration_state_<config>.json` файл `.migration.pause` — процесс перейдёт в паузу. Удалите файл для продолжения.
- **Потеря сети** — скрипт пытается переподключиться с экспоненциальной паузой (до `reconnect_max_wait` сек). После восстановления связи запустите `./run.sh run` снова — дубликатов не будет.

## Повторный запуск и обрывы

Состояние пишется в `migration_state_<config>.json` (Message-ID перенесённых писем, UID-кэш по папкам, UIDVALIDITY). Уже перенесённые письма при следующем запуске пропускаются: по UID-кэшу — без повторных FETCH-запросов к серверу, что сильно ускоряет возобновление на больших папках. Если скрипт упал или вы остановили его по Ctrl+C — просто запустите `./run.sh run` снова: он продолжит с того же места.

## Ручной запуск без run.sh

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

python imap_migrate.py -c config.yaml --list-folders
python imap_migrate.py -c config.yaml --dry-run
python imap_migrate.py -c config.yaml
```

## Параметры командной строки

| Параметр | Описание |
|----------|----------|
| `-c`, `--config` | Путь к конфигу (обязательно) |
| `-n`, `--dry-run` | Проверка без записи на целевой сервер |
| `-f`, `--folders` | Папки через запятую (иначе обрабатываются все) |
| `-v`, `--verbose` | Подробный лог (DEBUG) |
| `--verify` | После переноса сравнить число писем по папкам |
| `--list-folders` | Показать папки источника и выйти |

## Что лежит в проекте

- `imap_migrate.py` — основной скрипт
- `config.yaml.example` — пример конфига (скопируйте в `config.yaml` и отредактируйте)
- `config.yaml` — ваш конфиг с паролями (в git не попадает)
- `run.sh` — создание venv и запуск команд
- `requirements.txt` — зависимости (pyyaml, tqdm)
- `CHANGELOG.md` — история изменений по версиям
- При работе создаются: `.venv/`, `migration_state_<config>.json`, `migration_<config>.log`; при паузе по файлу — `.migration.pause` (в каталоге с state-файлом)

## Частые проблемы

**Ошибка входа (Login failed)** — для Яндекса и Gmail используйте пароль приложения. Проверьте, что в настройках почты включён доступ по IMAP. Для других провайдеров убедитесь в правильности логина и пароля.

**Обрывы соединения, таймауты** — можно увеличить `timeout` и `throttle` в конфиге; при нестабильной сети скрипт сам переподключается, после появления сети запустите его снова.

**В веб-интерфейсе у писем лишние теги** — нестандартные флаги вроде `$NotJunk` при переносе отбрасываются; при необходимости задайте `exclude_flags` в конфиге.

**Python не находится** — на macOS: `brew install python@3.12` или установка с python.org.

**Число писем не сходится при verify** — часто помогает повторный запуск `./run.sh run`, если в первый раз были единичные ошибки или обрывы.
